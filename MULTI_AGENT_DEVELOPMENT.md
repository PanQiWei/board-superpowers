# Multi-Agent Development Guide

> **Required reading** for anyone designing multi-agent / orchestration
> features in board-superpowers — most immediately the **Mode-2
> Consumer** path where a Producer agent spawns Consumer as a
> subagent. Hooks into the same dual-platform contract as
> `PLUGIN_DEVELOPMENT.md`: every primitive used by Mode-2 must work
> on **Claude Code** AND **OpenAI Codex CLI**, or be explicitly
> tagged as platform-only.

This document references the official documentation for both
products' multi-agent surfaces — subagent / Task / Agent tools, agent
teams, session reachback, and the lifecycle hooks that fire in those
contexts. Where docs are silent or the two platforms diverge, the
**Honest gaps** section flags it explicitly. Do not invent contracts.

**URL freshness:** all URLs verified **2026-04-26**. Re-verify when
modifying related code; a moved or broken canonical URL is a
load-bearing fact and must be patched in the PR that catches it.

The CC subagent / agent-team surfaces are evolving fast. The
`SendMessage` tool and the agent-teams feature are explicitly
flagged as **experimental** by Anthropic and gated behind
`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`. Treat any feature here
marked "experimental" as load-bearing-only for prototypes — do not
ship downstream-user-facing behavior on top without a stable-version
recheck.

## TL;DR — surface mapping

Multi-agent capability surface, CC ↔ Codex. **Bold** entries are the
ones the Mode-2 Consumer design depends on directly.

| Capability | Claude Code | Codex CLI | Portability |
|------------|-------------|-----------|-------------|
| **Spawn a child agent from inside a session** | `Agent` tool (renamed from `Task` in v2.1.63; `Task(...)` still works as alias) | `spawn_agents_on_csv` for batch + natural-language orchestration; **explicit user instruction required** | **Diverged** — different invocation models, different lifecycles |
| Mid-flight message to a running child | `SendMessage` tool — **experimental, agent-teams-gated** (`CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`) | Natural-language steering through `/agent` thread switch; no structured IPC tool | **Diverged** |
| Resume a stopped child / re-enter its context | `SendMessage` to a stopped subagent auto-resumes it; transcript at `~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl` | Not documented for subagents; top-level `codex exec resume <SESSION_ID>` resumes *interactive*-style sessions | **Diverged** — CC has it, Codex does not for child agents |
| Independent context window per child | Yes — subagent + agent-team teammates each have their own | Yes — subagents documented to address "context pollution" / "context rot" | **Portable** |
| Child can spawn further children | **No** — subagents cannot spawn subagents; agent-teams cannot nest teams | Not documented (treat as no) | **Portable (negative)** |
| Background / async child | `background: true` frontmatter or `Ctrl+B`; permission pre-approval at spawn time | Job control via `max_concurrency`, `agents.job_max_runtime_seconds` | **Diverged** semantics, both have *some* primitive |
| Forked child (inherits full conversation) | `/fork` command; experimental; `CLAUDE_CODE_FORK_SUBAGENT=1` | Not documented | **Claude-only** |
| Lifecycle hooks fired in child context | `SubagentStart`, `SubagentStop`, `TeammateIdle`, `TaskCreated`, `TaskCompleted`; fire in **parent** context with `agent_id` | Hooks fire on tool calls inside `codex exec`; subagent-context firing not specified | **Diverged** |
| Session-id reachback (parent reads child's transcript file) | `~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl`, retained per `cleanupPeriodDays` (default 30) | `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`; session id auto-generated by backend, embedded in JSONL | **Portable in spirit** — both have on-disk transcripts; lookup mechanics differ |
| Structured-return contract from child | `--output-format json` + `--json-schema` (Agent SDK / `claude -p`); `system/init` event reports session metadata | `--output-last-message <FILE>` + `--output-schema <JSON-Schema>`; UUID v7 session id in stream | **Portable** |
| Worktree isolation per child | `isolation: worktree` frontmatter on subagent; `EnterWorktree` / `ExitWorktree` tools (not available to subagents) | Not documented — Codex sandbox modes (`workspace-write`, etc.) are the closest analog | **Claude-only** |
| `AskUserQuestion` from inside a child | Available in foreground subagents (passes through to user); fails in background subagents | Not documented as a separate tool | **Diverged** |

## Claude Code

### Subagents (`Agent` tool)

- **Official docs**: <https://code.claude.com/docs/en/sub-agents>
- **Tool name**: `Agent` (renamed from `Task` in v2.1.63; `Task(...)`
  references still work as aliases). Use exact-string `Agent` in
  permission rules and `tools:` allowlists.
- **Definition**: subagents are Markdown files with YAML frontmatter
  under one of: managed settings, `--agents` CLI JSON, project
  `.claude/agents/`, user `~/.claude/agents/`, or plugin `agents/`
  directory. Five-level scope precedence (managed → CLI → project →
  user → plugin) with the higher level winning.
- **Frontmatter (required)**: `name`, `description`. Description is
  match behavior — Claude auto-delegates by matching the prompt
  against the description string.
- **Frontmatter (optional, multi-agent-relevant)**: `tools`,
  `disallowedTools`, `model`, `permissionMode`, `maxTurns`, `skills`,
  `mcpServers`, `hooks`, `memory`, `effort`, `background`,
  `isolation` (set to `worktree` for repo-isolated execution),
  `color`, `initialPrompt`.
- **Process model**: each subagent invocation creates a **new
  instance with fresh context**. Subagent inherits the parent's
  current working directory; `cd` inside the subagent does not
  persist between Bash calls. Use `isolation: worktree` for an
  isolated repo copy (auto-cleaned if the subagent makes no
  changes).
- **Hard limit**: subagents cannot spawn other subagents. The
  built-in `Plan` subagent exists *because* of this rule — it lets
  plan-mode gather context without recursion.
- **Permission inheritance**: subagent inherits permission context
  from parent. Parent `bypassPermissions` / `acceptEdits` /
  `auto` mode takes precedence and cannot be overridden.

#### Built-in subagents (always available)

| Name | Model | Tools | Purpose |
|------|-------|-------|---------|
| `Explore` | Haiku | Read-only | File discovery + codebase search |
| `Plan` | Inherits | Read-only | Plan-mode research (avoids recursion) |
| `general-purpose` | Inherits | All | Multi-step tasks needing both exploration + action |
| `statusline-setup` | Sonnet | (varies) | `/statusline` configuration |
| `Claude Code Guide` | Haiku | (varies) | In-product help |

#### Foreground vs background subagents

- **Foreground (default)**: blocks the parent. Permission prompts
  and `AskUserQuestion` calls pass through to the user.
- **Background**: runs concurrently. Permissions are **pre-approved
  at spawn time**; anything not approved is auto-denied at runtime.
  `AskUserQuestion` from a background subagent **fails** (the tool
  call errors but the subagent keeps running).
- Triggered via `background: true` frontmatter, natural-language
  ("run this in the background"), or `Ctrl+B` to push a running
  task to background.
- Disable globally: `CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1`.

### Subagent resumption + `SendMessage`

- **Tool name**: `SendMessage` — **only available** when
  `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` is set.
- **Two uses**:
  1. **Resume a stopped subagent** by passing its agent ID as the
     `to` field. Stopped subagents auto-resume in the background
     when sent a message; no new `Agent` invocation needed. Resumed
     subagents retain full conversation history (tool calls,
     reasoning, results).
  2. **Send a message to an agent-teams teammate** (see Agent teams
     below).
- **Agent ID lookup**: returned by Claude when the subagent
  completes; also discoverable in transcript files at
  `~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl`.
- **Persistence**: subagent transcripts persist *independently* of
  the main conversation — main-conversation compaction does not
  affect them. Cleanup follows the `cleanupPeriodDays` setting
  (default 30 days).

### Forked subagents (`/fork`)

- **Official docs**: <https://code.claude.com/docs/en/sub-agents#fork-the-current-conversation>
- **Status**: experimental, requires v2.1.117+ and
  `CLAUDE_CODE_FORK_SUBAGENT=1`.
- **Behavior**: a fork is a subagent that inherits the **entire
  conversation history**, system prompt, tools, and model from the
  parent — drops the input-isolation property. Output isolation is
  preserved: the fork's own tool calls do NOT leak back into the
  parent's context; only the final result returns.
- **Limitations**: works **only in interactive sessions** —
  disabled in non-interactive mode (which includes the **Agent
  SDK**). Forks cannot spawn further forks. With fork mode enabled,
  every subagent spawn runs in the background regardless of the
  `background` field.

### Agent teams

- **Official docs**: <https://code.claude.com/docs/en/agent-teams>
- **Status**: **experimental**, gated by
  `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`. Requires CC v2.1.32+.
- **Architecture**: lead session + N teammate sessions, each a
  **fully independent CC instance with its own context window**.
  Distinct from subagents: teammates can message *each other*
  directly via `SendMessage`, not just report back to the lead.
- **Created via tool calls**: `TeamCreate`, `TeamDelete` (both
  experimental-gated). Coordination via shared task list
  (`TaskCreate` / `TaskGet` / `TaskList` / `TaskUpdate` /
  `TaskCompleted` / `TaskCreated` hooks) and `SendMessage` mailbox.
- **Disk artifacts**:
  - `~/.claude/teams/{team-name}/config.json` — runtime state
    (session IDs, tmux pane IDs). **Do not edit by hand** — the
    runtime overwrites it on every state update.
  - `~/.claude/tasks/{team-name}/` — shared task list.
- **Display modes**: `in-process` (Shift+Down to cycle), `tmux` /
  iTerm2 split panes, or `auto` (default).
- **Permissions**: all teammates start with the lead's permission
  mode. Cannot set per-teammate modes at spawn time.
- **Subagent definitions are reusable** as teammate roles —
  reference by `name`. The body becomes additional system-prompt
  guidance; `tools` and `model` are honored. **`skills` and
  `mcpServers` frontmatter from the subagent definition are NOT
  applied** when running as a teammate (teammates load these from
  project + user settings instead).
- **Hard limits**: one team per lead session, no nested teams,
  lead is fixed for team lifetime, teammates cannot spawn
  teammates, in-process teammates do not survive `/resume` or
  `/rewind`.

### Lifecycle hooks in subagent / team context

- **Official docs**: <https://code.claude.com/docs/en/hooks>
- All five subagent / team hook events fire in the **parent**
  context. The payload includes `agent_id` (and `agent_type` for
  the subagent / teammate variants) so hooks can distinguish which
  child triggered them.

| Event | Matcher | Fires when | Block effect |
|-------|---------|------------|--------------|
| `SubagentStart` | `agent_type` | Subagent spawned via `Agent` tool | None — exit code 2 ignored, used for logging only |
| `SubagentStop` | `agent_type` | Subagent finishes | Exit 2 prevents stop (subagent continues) |
| `TeammateIdle` | none | Agent-teams teammate about to go idle | Exit 2 prevents idle (teammate continues) |
| `TaskCreated` | none | Task being created via `TaskCreate` | Exit 2 rolls back creation |
| `TaskCompleted` | none | Task being marked complete | Exit 2 prevents completion |

- **Frontmatter `Stop` hooks** declared in a subagent's SKILL.md /
  agent file are **automatically converted to `SubagentStop`** at
  runtime when the agent runs as a subagent.
- **Frontmatter hooks fire in two contexts**: when the agent runs
  as a subagent (via `Agent` tool / @-mention), AND when the agent
  runs as the main session (via `--agent`). Plugin subagents
  **cannot** declare `hooks`, `mcpServers`, or `permissionMode` —
  these fields are silently ignored when loaded from a plugin.
- **`Elicitation` / `ElicitationResult`**: fire during MCP tool
  execution when the server requests user input. Not subagent
  lifecycle events — they're MCP-elicitation events that happen to
  also flow through the hook system. The matcher is the MCP
  server name.

### `AskUserQuestion` inside a subagent

- Documented behavior: in foreground subagents, `AskUserQuestion`
  passes through to the user (parent terminal). In background
  subagents, the call **fails** (tool errors out) but the subagent
  keeps running. There is no explicit doc on whether the question
  reaches the *parent agent* vs the user directly — the docs say
  "passed through to you" (the user) for foreground; treat the
  background path as a non-channel.

### Agent SDK / non-interactive (`claude -p`)

- **Official docs**: <https://code.claude.com/docs/en/headless>
- **Subagent / fork limitations**: `/fork` is **disabled in
  non-interactive mode (which includes the Agent SDK)**. Subagent
  spawning via `Agent` tool itself is not explicitly disabled, but
  background-mode pre-approval semantics still apply.
- **Structured outputs**: `--output-format json` + `--json-schema`
  produces validated JSON in the `structured_output` field.
  `--output-format stream-json` + `--verbose` +
  `--include-partial-messages` emits NDJSON event stream.
- **Session id reachback**: `claude -p ... --output-format json |
  jq -r '.session_id'` captures the session id; `--resume <id>`
  re-enters it. `--continue` resumes the most recent.
- **`--bare` mode**: skips auto-discovery of hooks, skills,
  plugins, MCP servers, auto memory, and CLAUDE.md. Recommended
  for SDK / scripted calls; will become the default for `-p` in a
  future release.

### Worktree integration

- `isolation: worktree` frontmatter on a subagent gives it an
  isolated repo copy; auto-cleaned if no changes made.
- `EnterWorktree` / `ExitWorktree` tools manage worktrees in the
  main session; **not available to subagents** themselves (per the
  tool reference).
- `WorktreeCreate` / `WorktreeRemove` hook events fire in the main
  session when worktrees are created / removed. Plugin contract
  for these events is not documented beyond their existence in the
  hook list.

## OpenAI Codex CLI

### Subagents

- **Official docs**: <https://developers.openai.com/codex/subagents>
  + <https://developers.openai.com/codex/concepts/subagents>
- **Spawn requires explicit user instruction**. From the docs:
  "Codex doesn't spawn subagents automatically, and it should only
  use subagents when you explicitly ask." Natural-language phrases
  like "spawn one agent per X, wait for all of them, summarize"
  trigger the orchestration layer.
- **Defined via `[agents]` section of `config.toml`** plus
  per-role files under `.codex/agents/` (project) or
  `~/.codex/agents/` (personal). Both the `[agents]` section and
  the per-role file schema are fully documented — see
  **`[agents]` config schema** below for the canonical fields.
- **Use cases the docs explicitly recommend**: read-heavy work
  (exploration, testing, log analysis), parallel review with
  distinct lenses (security / maintainability / coverage), and
  large document review. Docs explicitly **warn against parallel
  write-heavy workflows** ("agents editing code at once can create
  conflicts").
- **Tools the model uses to spawn**:
  - `spawn_agents_on_csv` — batch tool. Parameters: `csv_path`,
    `instruction` (with `{column_name}` placeholders), `id_column`,
    `output_schema`, `max_concurrency`,
    `max_runtime_seconds`. Workers must call
    `report_agent_job_result` exactly once.
  - **No documented manual-spawn API**. Manual spawning happens
    through natural-language requests; no explicit one-shot tool
    name is listed.
- **Lifecycle**: spawn → steer (via natural language) → stop /
  close. The `/agent` slash command in the CLI switches the
  active agent thread for inspection. Per-worker timeout
  defaulted by `agents.job_max_runtime_seconds`; per-call
  overrides win.
- **Result collection**: "Codex waits until all requested results
  are available, then returns a consolidated response." For CSV
  batch, results export to CSV with `job_id` and `status`.
- **Token cost is documented as a load-bearing trade-off**:
  "subagent workflows consume more tokens than comparable
  single-agent runs."
- **Resume**: not documented for subagents. Top-level `codex exec
  resume <SESSION_ID>` resumes *parent* sessions, not children.
  Treat subagent-resume as **not supported on Codex** until a
  primary source says otherwise.
- **IPC primitives**: not documented. Communication is mediated
  by the orchestration layer (parent invokes, parent waits,
  parent receives consolidated result). No documented analogue to
  CC's `SendMessage`.

### `[agents]` config schema

- **Official docs**: <https://developers.openai.com/codex/config-reference>
  + canonical TOML sample at <https://developers.openai.com/codex/config-sample>
  + machine-readable JSON schema at
  <https://github.com/openai/codex/blob/main/codex-rs/core/config.schema.json>
  (`AgentsToml` and `AgentRoleToml` definitions).
- The `[agents]` section in `~/.codex/config.toml` (or per-repo
  `<repo>/.codex/config.toml`) controls subagent threading,
  recursion, and worker timeouts. Per-role definitions live as
  sub-tables `[agents.<role>]`, optionally pointing at a
  standalone TOML file via `config_file`.

#### `[agents]` global keys

| Key | Type | Default | Source | Notes |
|-----|------|---------|--------|-------|
| `max_threads` | `uint` (min 1) | `6` | docs + JSON schema | "Maximum concurrently open agent threads." When unset in JSON-schema terms, no cap is enforced; the docs site quotes 6 as the default — treat 6 as the effective ceiling. |
| `max_depth` | `int32` (min 1) | `1` | docs + JSON schema | "Maximum nested spawn depth. Root session starts at depth 0." Default `1` allows direct child agents; deeper nesting is rejected. |
| `job_max_runtime_seconds` | `uint64` (min 1) | `1800` | docs + JSON schema | Per-worker timeout for `spawn_agents_on_csv` jobs. Schema marks the field as optional (`Option<u64>` in `codex-rs/core`); when unset, `spawn_agents_on_csv` falls back to `1800` seconds per worker. |
| `interrupt_message` | `bool` | `true` | JSON schema only | "Whether to record a model-visible message when an agent turn is interrupted." Documented in the JSON schema but not on the prose docs page — treat as stable but cite the schema, not the rendered docs. |

Canonical sample (verbatim from
<https://developers.openai.com/codex/config-sample>):

```toml
[agents]
# Maximum concurrently open agent threads. Default: 6
# max_threads = 6
# Maximum nested spawn depth. Root session starts at depth 0. Default: 1
# max_depth = 1
# Default timeout per worker for spawn_agents_on_csv jobs. When unset, the tool defaults to 1800 seconds.
# job_max_runtime_seconds = 1800

# [agents.reviewer]
# description = "Find correctness, security, and test risks in code."
# config_file = "./agents/reviewer.toml"  # relative to the config.toml that defines it
# nickname_candidates = ["Athena", "Ada"]
```

#### `[agents.<role>]` per-role keys (`AgentRoleToml`)

| Key | Type | Required | Notes |
|-----|------|----------|-------|
| `description` | string | required (unless supplied by the referenced role file) | "Human-facing role documentation used in spawn tool guidance." Drives the spawn-tool description the model sees. |
| `config_file` | path (`AbsolutePathBuf`) | optional | "Path to a role-specific config layer. Relative paths are resolved relative to the `config.toml` that defines them." |
| `nickname_candidates` | string[] | optional | "Candidate nicknames for agents spawned with this role." Codex picks display names from this pool. |

#### Standalone agent role file schema (`.codex/agents/<role>.toml` or referenced via `config_file`)

Documented at <https://developers.openai.com/codex/subagents>.

| Key | Type | Required | Notes |
|-----|------|----------|-------|
| `name` | string | required | Spawn identifier the model uses. |
| `description` | string | required | "Human-facing guidance for when Codex should use this agent." |
| `developer_instructions` | string (multi-line) | required | "Core instructions that define the agent's behavior." Becomes the role's system prompt. |
| `nickname_candidates` | string[] | optional | Same as `[agents.<role>].nickname_candidates`. |
| `model` | string | optional | Override parent session model, e.g., `gpt-5.5`, `gpt-5.4`, `gpt-5.4-mini`, `gpt-5.3-codex-spark`. |
| `model_reasoning_effort` | string | optional | `low` / `medium` / `high`. |
| `sandbox_mode` | string | optional | `read-only` / `workspace-write` / `danger-full-access` — same vocabulary as `codex exec --sandbox`. |
| `[mcp_servers.<name>]` | table | optional | Standard MCP server config (`url`, `command`, `args`, `startup_timeout_sec`, etc.) — scoped to this role. |
| `[[skills.config]]` | array of tables | optional | Per-role skill toggles, e.g., `path = "/abs/path/SKILL.md"`, `enabled = false`. |

Canonical example (verbatim from
<https://developers.openai.com/codex/subagents>):

```toml
# .codex/agents/reviewer.toml
name = "reviewer"
description = "PR reviewer focused on correctness, security, and missing tests."
model = "gpt-5.4"
model_reasoning_effort = "high"
sandbox_mode = "read-only"
developer_instructions = """
Review code like an owner.
Prioritize correctness, security, behavior regressions, and missing test coverage.
Lead with concrete findings, include reproduction steps when possible, and avoid style-only comments unless they hide a real bug.
"""
nickname_candidates = ["Atlas", "Delta", "Echo"]
```

Built-in role names (always available without a `[agents.<role>]`
declaration): `default`, `worker`, `explorer`. Override behavior by
declaring a same-named `[agents.<role>]` block with a custom
`config_file`.

### Sessions and `codex exec`

- **Official docs**: <https://developers.openai.com/codex/cli/reference>
  + <https://developers.openai.com/codex/noninteractive>
- **Storage location**:
  `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl` on macOS / Linux.
  Each session is one JSONL file with the full transcript, tool
  calls, and token usage. Path layout verified 2026-04-26 via
  primary docs and the `openai/codex` repo discussions.
- **Session-id format**: UUID v7. Generated by the OpenAI backend
  when the session starts; not user-settable. The id is embedded
  inside the rollout JSONL — renaming the file does not change
  the id `codex exec resume` looks up.
- **Resume mechanics**: `codex exec resume [--last|--all|<SESSION_ID>]`.
  `--last` from current cwd, `--all` to consider sessions outside
  cwd, or pass an explicit SESSION_ID. Each resumed run preserves
  the original transcript, plan history, and approvals. Resume
  works across processes / machines as long as the rollout file is
  reachable at the same path. Cross-user / cross-machine resume is
  **not documented** as a supported pattern (treat with caution).
- **Ephemeral mode**: `--ephemeral` runs without persisting
  rollout files. Trade-off: no resume, no later inspection.
- **`--output-last-message <FILE>`**: writes the assistant's final
  message to the file AND to stdout. Schema not constrained
  beyond "the assistant's final message" — combine with
  `--output-schema` for guarantees.
- **`--output-schema <JSON-Schema>`**: validates final response
  against the schema. Codex tool output is validated against this
  if provided.
- **`--json` event stream**: NDJSON. Documented event types:
  `thread.started`, `turn.started`, `turn.completed`,
  `turn.failed`, `item.*`, `error`. `item.completed` carries
  fields: `id`, `type` (`agent_message`, `command_execution`,
  file changes), `text`, `usage` (`input_tokens`,
  `cached_input_tokens`, `output_tokens`).

### Hooks in subagent / non-interactive context

- **Official docs**: <https://developers.openai.com/codex/hooks>
- The 6 documented events (`SessionStart`, `PreToolUse`,
  `PermissionRequest`, `PostToolUse`, `UserPromptSubmit`, `Stop`)
  do **not** specify whether they fire inside `codex exec` vs
  interactive `codex`, nor whether they fire inside a Codex
  subagent's context. Both are honest gaps as of 2026-04-26 — see
  Honest gaps section.
- **Trust gate**: project-local hooks (`<repo>/.codex/hooks.json`,
  `<repo>/.codex/config.toml`) load **only when the project
  `.codex/` layer is trusted**. User and system hooks always
  load. This trust gate is critical for downstream-user safety
  when their project ships hooks.

### Slash commands related to multi-agent

| Command | Effect |
|---------|--------|
| `/agent` | Switch active agent thread (inspect / continue work in a spawned subagent thread) |
| `/agents` | Browse / manage agents (parallel to CC's `/agents`) |
| `/resume` | Resume a saved interactive session via picker |
| `/new` | Start a fresh conversation in current CLI session |
| `/fork` | Clone the current conversation into a new parallel thread |
| `/status` | Show session config + token usage; lets you copy session id |

### Agents SDK (separate from CLI subagents)

- **Official docs**: <https://developers.openai.com/codex/guides/agents-sdk>
- This is a different surface from CLI subagents. It runs Codex
  as a long-lived MCP server and orchestrates **multiple top-level
  agents via handoffs** (sequential), not parent-child subagent
  spawning.
- Session continuity via `threadId` in
  `structuredContent.threadId` from `tools/call` responses.
- Out of scope for board-superpowers Mode-2 today — board
  Producer / Consumer is a CLI-level concern. Listed here for
  completeness.

## What this means for board-superpowers

The Mode-2 Consumer design (Producer agent spawns Consumer as a
subagent) closes under the constraints set by ADR-0007 and
ADR-0006. Concrete implications, in priority order:

### 1. Mode-2 must close under C-PLUGIN-1 (no in-memory cross-session IPC)

CC's `SendMessage` looks like a counter-example — it IS in-memory
mid-flight communication between parent and child. But it is:

- **Experimental** and gated by `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`.
- **Claude-Code-only** — no Codex equivalent (`/agent` thread
  switch is interactive-only and human-driven).

Therefore Mode-2 **must NOT depend on `SendMessage` for
correctness**. The portable signal channel between Producer and
Consumer remains the GitHub board (C-PLUGIN-1 workaround (a)) +
session-id reachback via on-disk transcripts (workaround (b)).
`SendMessage` may be used as a *latency optimization on the CC
fast path* if both are documented as platform-conditional; it
must never be the only signal.

### 2. Session-id reachback is portable but mechanically diverged

Both platforms persist child-session transcripts to disk:

- **CC**: `~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl`,
  retained per `cleanupPeriodDays` (default 30).
- **Codex**: `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`,
  session id is UUID v7 embedded in the file.

Mode-2's "Producer reads what Consumer did" path can use this on
both platforms. The lookup mechanics differ enough that the
abstraction belongs in `scripts/lib/common.sh` so callers don't
need to know which platform they're on. **Implementation
note**: parsing JSONL transcripts is fragile and not a stable
contract — prefer architecting Producer to read board-level
artifacts (PR body, claim-branch markers) where possible.

### 3. Mode-2 dispatch is autonomous — match D-AUTONOMY-1 row 13

Row 13 of ADR-0006's permission matrix marks **Dispatch Consumer
session = A** (autonomous). The technical reading of "dispatch":

- **CC fast path**: Producer calls `Agent` tool with the Consumer
  agent definition. Subagent inherits the Producer's permission
  context (subject to the precedence rules above). Audit-log entry
  written per row 13.
- **Codex fast path**: Producer issues a natural-language
  spawn-and-collect instruction (or `spawn_agents_on_csv` for
  the F-5 overnight batch case). Concurrency cap from C-PLUGIN-3
  maps to `max_concurrency` directly.

Mode-2 dispatch under autonomous-with-transparency is **clean on
both platforms** — neither requires an interactive approval to
spawn a child.

### 4. Subagents cannot spawn subagents — Mode-2 topology is one-deep

CC: explicit hard rule. Codex: not documented as a hard rule but
no documented mechanism for nested spawning either. Treat as
universal: **Mode-2 Consumer cannot spawn a sub-Consumer**. If
the work needs decomposition, the Consumer either escalates back
to the Producer (via board card body update) or completes the
unit it was given. No nested topologies.

This also limits the F-5 overnight-batch design: a single
Producer can spawn N Consumers, but the Consumers themselves are
flat workers. Anything that needs hierarchy must be expressed as
N cards, not as nested subagents.

### 5. C-PLUGIN-2 (no daemon) is reinforced by both platforms' subagent semantics

Background CC subagents are still session-scoped — they don't
outlive the parent session. Codex subagents are bounded by
`agents.job_max_runtime_seconds`. **Neither platform gives us a
daemon.** Mode-2 cannot rely on a "long-running Consumer" model;
each Consumer dispatch is a finite, bounded run that closes when
the PR opens or when a hard timeout elapses. Preflight piggyback
remains the only awareness pattern.

### 6. AskUserQuestion semantics differ — design Consumer to NOT need user input

CC foreground subagents pass `AskUserQuestion` through to the
user; background subagents fail it. Codex has no documented
equivalent. **Design rule**: Consumer must never depend on
asking the architect a question mid-flight. If clarification is
needed, the Consumer escalates by:

- (a) Updating the card body or posting a comment on the PR /
  card with the question (board-mediated, C-PLUGIN-1 workaround
  (a)).
- (b) Marking the card status as `Blocked` (D-AUTONOMY-1 row 6 is
  R-class — Producer must surface to architect on next prompt
  via preflight piggyback).

The Consumer never blocks waiting for synchronous architect
input. This rule is forced by C-PLUGIN-2 and reinforced by the
divergent `AskUserQuestion` semantics.

### 7. Worktree isolation is Claude-only at the platform level

CC: `isolation: worktree` on subagent + `EnterWorktree` /
`ExitWorktree` tools (the latter not available to subagents).
Codex: no documented worktree primitive — closest analog is
`workspace-write` sandbox, which gates writes but does not
isolate the working tree.

board-superpowers already owns its own worktree mechanism
(`scripts/claim-card.sh` creates a worktree at
`$HOME/.config/superpowers/worktrees/<project>/<branch>`). Mode-2
should keep using `claim-card.sh` for worktree isolation on both
platforms — do NOT delegate to CC's `isolation: worktree`
because that breaks the cross-platform portability promise.

### 8. Lifecycle hooks for Mode-2 should fire in Producer context

CC's `SubagentStart` / `SubagentStop` fire in the **parent**
context with `agent_id` payload. board-superpowers' Producer-side
"a Consumer just finished" preflight piggyback can hook
`SubagentStop` on CC to short-circuit the lookup (instead of
polling the board). This is a **CC-only optimization**; the
portable fallback is GitHub-state polling on the next architect
prompt. Codex hook coverage of subagent context is not
documented (Honest gaps).

### 9. Tools-list portability — name `Agent` and `SendMessage` as Claude-only

When a board-superpowers SKILL.md needs to reference tool names
to be executed, follow `PLUGIN_DEVELOPMENT.md` §8: tool-portable
prose for cross-platform skills. `Agent` and `SendMessage` are
Claude-Code-only tool names. Reserve them for skill paths
explicitly tagged Claude-Code-only.

### 10. Experimental flags are a runtime concern — preflight must check them

If Mode-2 ships features dependent on `SendMessage` or agent
teams, the SKILL preflight (Layer 2 of the dep-alert system in
`AGENTS.md`) must check for `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`
before invoking that path. This is a new check class — add it to
`scripts/check-deps.sh` if Mode-2 lands a `SendMessage`-dependent
optimization.

## Honest gaps in official docs

The following are **not** documented as stable contracts on at
least one platform; design accordingly. None of these block
Mode-2 v1, but each is a known unknown that should be re-checked
periodically.

- **CC subagent process model.** Docs describe "own context
  window" and on-disk transcript files but do NOT explicitly state
  whether subagents are separate OS processes or threads. Treat
  as opaque; do not depend on either model.
- **CC `SendMessage` schema.** Listed in the tools-reference table
  but the full input schema is not in the public reference page.
  Currently observable only via tool-call traces. Treat the tool
  as an in-context affordance for the model, not a stable
  programmatic contract.
- **Codex hook events × non-interactive mode.** The 6 documented
  events don't specify whether they fire inside `codex exec`
  (vs interactive `codex`). Empirical testing required if Mode-2
  Codex variant relies on hook firing.
- **Codex hook events × subagent context.** Same gap — the docs do
  not specify whether hooks fire inside a Codex-spawned
  subagent's context. Treat as "no" until proven otherwise.
- **Codex subagent resume.** No documented mechanism. Treat
  Codex-side Mode-2 as **non-resumable** — each Consumer
  dispatch is a one-shot until proven otherwise.
- **Codex IPC primitives between parent and subagent.** None
  documented. Communication is opaque, mediated by the
  orchestration layer. Mode-2 Codex variant should treat
  parent ↔ child as one-shot RPC: spawn → wait → consolidated
  result.
- **Codex agent-team analogue.** None documented. CC has agent
  teams; Codex has subagents-with-orchestration but no documented
  peer-to-peer messaging primitive among siblings.
- **Cross-user / cross-machine session resume.** Both platforms
  store transcripts on local disk. Neither documents resume from
  a transcript file moved to another user / machine. Treat as
  unsupported.
- **`SendMessage` and `TeamCreate` / `TeamDelete` stability.**
  Explicitly experimental, gated by env var. May change without
  semver. Don't ship downstream-user features that hard-depend on
  the current shape.
- **CC `--bare` mode + subagents.** `--bare` skips skills /
  plugins / hooks discovery. Behavior of subagent spawning under
  `--bare` is not explicitly documented. If Mode-2 dispatches
  Consumer in CI via `claude -p --bare`, validate empirically.
- **Codex tool name for manual subagent spawn.** Docs name
  `spawn_agents_on_csv` for batch but list no manual-spawn tool.
  Manual spawn is "natural-language driven." Treat as not having
  a stable tool-call API for one-off spawns.

## Maintenance discipline for this doc

- All URLs verified **2026-04-26**. **Re-verify when modifying
  related code.** A broken or moved canonical URL is a
  load-bearing fact and must be patched in this PR, not deferred.
- When a new multi-agent surface lands in either product
  (especially: `SendMessage` graduates from experimental, Codex
  ships peer-to-peer subagent messaging, Codex documents hook
  firing in subagent context), add a section here **before** the
  first board-superpowers feature uses it. This doc is leading,
  not trailing.
- When a contract from this doc changes (a tool renamed, an
  experimental flag removed, a new lifecycle event), update both
  the section here AND the dependent code in board-superpowers in
  the same PR.
- When closing a Honest-gap (e.g., empirically establishing that
  Codex hooks fire in `codex exec`), promote it from this section
  to the relevant platform section with a note: "Empirically
  verified <date>; not in primary docs." Link to the test or
  reproducer.
- This doc is referenced **by name** from `AGENTS.md`'s
  "Cross-cutting reference docs" table — not loaded with
  `@`-prefix — so it does not ride into every session's
  context. The `skills/AGENTS.md` per-directory contract
  sends agents here when designing orchestration where one
  agent spawns another (Producer → Consumer subagent,
  agent-teams, fan-out / fan-in pipelines).

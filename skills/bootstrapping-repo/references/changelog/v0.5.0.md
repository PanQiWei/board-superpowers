# changelog — board-superpowers v0.5.0

This changelog file is consumed by `bootstrapping-repo` when the lifecycle diff detects a generation bump on the M1 changelog-surface stage (agentic; surfaces migration highlights to the architect). Surface this content inline when upgrading from any v0.4.x version to v0.5.0 on this host or repo.

Full release notes: `https://github.com/PanQiWei/board-superpowers/releases/tag/v0.5.0`

## What's new for the host

v0.5.0 introduces the **setup-stages system** — a declarative mechanism that unifies first-time setup and plugin-upgrade reconvergence into a single skill and a single registry. Architects who previously ran `bootstrap-host.sh` and `bootstrap-project.sh` directly will now see `bootstrapping-repo` drive those operations stage-by-stage with lifecycle tracking.

Key changes:

- `~/.board-superpowers/manifest.yml` → replaced by `~/.board-superpowers/settings.yml § stages_completed`. The manifest file is no longer written; the M1 stage's completion record is the authoritative host bootstrap marker.
- `~/.board-superpowers/repos/<id>/state.yml` → replaced by `~/.board-superpowers/repos/<repo-identity>/settings.yml § stages_completed`. Same host-local storage, new schema.
- `<repo>/.board-superpowers/config.yml` and `config.local.yml` → the `config.yml` module config sections migrate into `<repo>/.board-superpowers/settings.yml § modules.*`. Clone-local values migrate into `<repo>/.board-superpowers/settings.local.yml`.

## What's new that affects every repo

1. **`operating-kanban` atomic skill** (shipped v0.5.0) — all board reads in M3 stages now route through `board-superpowers:operating-kanban` (ADR-0027) instead of inline `gh` calls. This decouples M3 stage executors from the active projection's CLI syntax.

2. **Kanban projection selection** (M10 stage) — a new agentic stage `m10.repo.choose-kanban-projection` prompts the architect to select the active kanban backend. In v0.5.0 only `github-project-v2` is available; `linear` and others land in v1.1.

3. **Four partitioned settings files** — the single `config.yml` / `state.yml` pair is replaced by four files with clear locality semantics. See `bootstrapping-repo/SKILL.md § On-disk settings layout` for the schema.

4. **Lifecycle vocabulary** — the six stage states (`pending / applied / drifted / blocked / failed / not-applicable`) are now stable. Prior informal vocabulary (`manifest_exists`, `state_completed`, etc.) is retired.

5. **Action_id catalog expanded** — bootstrap actions now use the 200-208 range with outbox (`--mode bootstrap-pending`) to support audit before the M4 DDL stage applies. See `classifying-actions/references/action-id-catalog.md`.

## Migration from v0.4.x

### Automatic migration (no architect action required)

When you open the first session in a v0.4.x repo after installing v0.5.0:

- The hook detects the absent v0.5.0 settings files → emits `INVOKE: bootstrapping-repo`.
- `bootstrapping-repo` runs the lifecycle diff → all stages are `pending`.
- Stages that were previously completed in the v0.4.x schema are re-applied (most are idempotent — no visible changes on a correctly-bootstrapped repo).

### Stages with architect interaction

M10 (kanban projection choice) is new in v0.5.0. You will be prompted once per repo:

```
Which kanban backend should this repo use?
1. github-project-v2  (GitHub Project V2 — default)
2. none               (No kanban — audit-log-only mode)
```

For existing repos that were already using GitHub Project V2, answer `1` (github-project-v2). The M3 stages then re-verify labels and Status field options (idempotent — no changes if the board is already correct).

### Files to clean up (optional)

The following files are no longer written by v0.5.0 and can be removed after confirming the new settings files exist:

- `~/.board-superpowers/manifest.yml`
- `~/.board-superpowers/repos/<normalized>/state.yml`
- `<repo>/.board-superpowers/config.yml`
- `<repo>/.board-superpowers/config.local.yml`

Do not delete these until after the M1 host-shared `settings.yml` and repo `settings.yml` exist and show all stages as `applied`.

## Breaking changes

- **`config.yml` and `config.local.yml` schema is retired.** Any hand-edits to `wip_limit` or `audit_db_url` in those files must be re-applied to the new settings files after the M4 / M8 stages apply. The migration flow prompts you on any detected divergence.
- **`bootstrap-host.sh` and `bootstrap-project.sh` are superseded.** The SKILL drives those scripts internally. Running them directly bypasses lifecycle tracking and may leave stages in an inconsistent state. Prefer the SKILL.
- **`migrating-repo-version` skill is not present in v0.5.0.** Migration is now handled by the `bootstrapping-repo` SKILL's lifecycle diff (new stages are `pending`; generation bumps are `drifted`). No separate migration step is needed.

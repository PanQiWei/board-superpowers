# action_id catalog — what each row means in practice

This file complements `matrix.md` by describing each row in prose, with
example actions that map to it. Use this when you have an action
description but aren't sure which row it lives in.

## Producer rows (1-14)

- **1: Create cards (decomposition output)**
  - Examples: `gh issue create` from a design doc; intake routine
    spawning a card from a brainstorm.
  - Excludes: editing an existing card's body (that's row 2).

- **2: Edit card body**
  - Examples: refining acceptance criteria; adding a note about a
    blocker; correcting a typo in the goal.
  - Excludes: changing status (row 5 / 6) or labels (row 11).

- **3: Split card**
  - Examples: a card grew too big mid-implementation; architect
    decides to split into 2-3 smaller cards.
  - Excludes: creating new cards from a fresh design doc (row 1).

- **4: Update CLAUDE.md / AGENTS.md**
  - Examples: adding a new doctrine paragraph; updating tech-stack
    list; correcting a routing block.

- **5: Backlog → Ready transition**
  - Examples: marking a card as Ready after intake validates the
    Acceptance Criteria pass INVEST.
  - Excludes: Ready → In Progress (Consumer's claim, row 100).

- **6: In Progress → Blocked transition**
  - Examples: Consumer hits a hard external dependency.

- **7: Close stale card**
  - Examples: closing a card that's been in Backlog for 6 months
    without progress.

- **8: Cancel claim**
  - Examples: a Consumer abandons a card mid-implementation;
    architect releases the claim to make it Ready again.

- **9: Adjust WIP limit**
  - Examples: changing `wip_limit` in `config.local.yml`.

- **10: Modify config.yml or config.local.yml**
  - Examples: editing the project number, adding an autonomy override,
    changing the base branch.

- **11: Extend GitHub Project fields**
  - Examples: adding a new label; adding a new Status option (e.g.,
    "Suspended").

- **12: Auto-merge PR**
  - Examples: gh pr merge with automation flags. Hard-floor blocked
    for Consumer (Producer-only).

- **13: Dispatch Consumer session**
  - Examples: spawning a Consumer subagent in Mode-2 overnight
    batch; manually invoking a Consumer in Mode-1.

- **14: Auto-trigger retro / weekly report**
  - Examples: cadence-driven retro firing on milestone close;
    weekly report generated by preflight piggyback.

## Consumer rows (100-113)

- **100: Claim card**
  - The 4-step transaction: Status flip + worktree create + branch
    create + branch push. Fires once per claim.

- **101: Surface (propose-and-suspend)**
  - Consumer hits a design decision point and surfaces it back to the
    architect via card thread comment.

- **102: Terminate — success**
  - PR merged, claim resolved, worktree cleaned up.

- **103: Terminate — failure**
  - Card moved to Blocked, claim released, worktree preserved for
    human takeover.

- **104: Retro Notes write**
  - Initial Retro Notes draft at PR submit + post-merge supplement.

- **105-111: Review-cycle responses**
  - Each sub-action that the Consumer takes during PR review:
    direct fix (105), re-delegation (106), verification chain (107),
    cross-platform review (108), QA pass (109), security audit (110),
    cycle completion (111).

- **112: PR-submit pre-flight — card body sync**
  - Just before opening the PR (after verification chain passes), the
    Consumer toggles every acceptance criterion checkbox in the card
    body from `[ ]` to `[x]` (or `[!]` for cleanly-deferred items
    with a one-line reason). May also append an implementation-summary
    paragraph if the card's Notes section calls for one.
  - Why A-class: the toggles reflect work that has just been verified
    in the previous step; rejecting at toggle-time would re-open
    decisions already settled. The audit row records the before/after
    sha256 of the card body and the count of toggled items.

- **113: Post-merge cleanup**
  - Once the PR is merged: remove the local worktree
    (`git worktree remove`), delete the local `claim/<N>-<slug>`
    branch (the remote was deleted by the merge), and write the
    terminal close audit row. Card status flip to `Done` is the
    GitHub webhook's responsibility, not this skill's; if the
    webhook lags >30s the Consumer surfaces it as a known issue
    rather than racing to flip Status manually.
  - Why A-class: post-merge cleanup is reversible (the worktree can
    be re-created from `origin/main`) and follows a settled pre-
    condition (PR is merged). When run via cron, the cron itself is
    R-class (architect opted-in via `auto_post_merge_cleanup` in
    config), so individual cleanup runs inside the cron stay A.
